Operators Lab

Operators are a way to create custom controllers in OpenShift and Kubernetes. They use the concept of Custom Resource Definitions to extend the Kubernetes API. The Operator then watches for custom resource objects to be created - and when it sees a custom resource being created it creates the application based on the parameters in the custom resource object.

Operators are usually written in the Go programming language (just like Kubernetes or OpenShift). This means that the Kubernetes API is used to create and monitor objects in the cluster directly.

Since this is not a very developer friendly experience Red Hat has provided a Software Development Kit that significantly simplifies the creation of Operators. The SDK handles all the core logic while the developer only needs to fill in the actual business logic.

Red Hat also provides to other capabilities in the SDK to create operators from both Ansible Roles or Playbooks and from Helm Charts.

In this lab you are using the Ansible Operator SDK to create an Operator using two Ansible Roles and a Playbook that sets up a PostgreSQL Database and a Gogs Server that uses that database. Unlike in templates this operator can fully configure the Gogs Server since it can query the route that has been created before adding it to the Gogs configuration.

See the Ansible Operator Documentation at https://github.com/operator-framework/operator-sdk/blob/v0.6.0/doc/ansible/user-guide.md for more information.

See the Helm Operator Kit at https://github.com/operator-framework/operator-sdk/blob/v0.6.0/doc/helm/user-guide.md for more information about Helm Charts. Note that Helm Charts need to satisfy the same requirements as regular OpenShift applications (e.g. pods do not run as root) in order to run successfully on OpenShift.

Goals

  • Create an Operator from Ansible Roles using the Ansible Operator SDK

  • Deploy the Operator to OpenShift

  • Create a Gogs Server using a Custom Resource

  • Create a second Gogs Server

  • Examine the created objects

  • Delete the Gogs Servers

  • Delete the Operator

2. Install the Operator SDK

The first step to building an Operator is to install the Operator SDK into your Client VM.

  1. Download the Operator SDK binary into /usr/local/bin:

    This lab uses version v0.6.0 because the Operator SDK changes quite frequently and you need to use a well-defined snapshot of the repository to guarantee a smooth lab.

    sudo wget https://github.com/operator-framework/operator-sdk/releases/download/v0.6.0/operator-sdk-v0.6.0-x86_64-linux-gnu -O /usr/local/bin/operator-sdk
sudo chmod +x /usr/local/bin/operator-sdk

  2. Add the directory /usr/local/bin to your PATH and save the PATH for future logins:

    export PATH=$PATH:/usr/local/bin
echo "export PATH=$PATH:/usr/local/bin" >> $HOME/.bashrc

  3. Now your Operator SDK is ready to be used.

  4. There is a repository that someone else built that includes Ansible roles to set up a PostgreSQL database and a Gogs Server on top of OpenShift. The repository also contains an Ansible playbook that calls these roles to create a Gogs Server backed by a PostgreSQL database in a project on OpenShift.

    You will now use the Ansible Operator to convert these roles into an Operator that can create Gogs installations by simple requesting a Custom Gogs Resource.

  5. Clone the repository with the roles

    cd $HOME
git clone https://github.com/redhat-gpte-devopsautomation/ansible-operator-roles
cd ansible-operator-roles
git checkout v0.6.0
cd $HOME

  6. Examine the Playbook:

    cat $HOME/ansible-operator-roles/playbooks/gogs.yaml

  7. If you want to you can also examine the two roles (and the default values for their different variables):

    cat $HOME/ansible-operator-roles/roles/postgresql-ocp/tasks/main.yml
cat $HOME/ansible-operator-roles/roles/postgresql-ocp/defaults/main.yml
cat $HOME/ansible-operator-roles/roles/gogs-ocp/tasks/main.yml
cat $HOME/ansible-operator-roles/roles/gogs-ocp/defaults/main.yml

  8. From your home directory use the operator-sdk binary to create a new Ansible Operator skeleton (you can use the Ansible Operator User’s Guide at https://github.com/operator-framework/operator-sdk/blob/v0.6.0/doc/ansible/user-guide.md for reference). Use the following values (if you don’t your operator will not work in our shared cluster):

    • gogs-operator as the operator name

    • gpte.opentlc.com/v1alpha1 as the API version

    • Gogs as the Kind

    • You may use --skip-git-init as an additional flag to skip creation of a Git Repo

      Make sure to pick Ansible as the operator type. Also make sure to include the option to use a playbook and not just roles.
      If you messed up the the creation you can just delete the directory (gogs-operator) and try again.

  9. The operator-sdk creates a skeleton playbook and a skeleton roles directory. But you already have a playbook and roles from the repository you cloned earlier.

  10. Switch into the directory and replace the playbook and roles directory with the ones you downloaded.

    cd $HOME/gogs-operator
rm -rf roles playbook.yml
mkdir roles
cp -R $HOME/ansible-operator-roles/roles/postgresql-ocp ./roles
cp -R $HOME/ansible-operator-roles/roles/gogs-ocp ./roles
cp $HOME/ansible-operator-roles/playbooks/gogs.yaml ./playbook.yml

  11. Examine the file watches.yaml. This is the main configuration for the Ansible Operator.

    cat ./watches.yaml
    Sample Output
    ---
- version: v1alpha1
  group: gpte.opentlc.com
  kind: Gogs
  playbook: /opt/ansible/playbook.yml

    You will notice that the group, version and kind came from the command you used to create the skeleton. The Playbook uses the playbook.yml filename. You will see how the file ends up in the location /opt/ansible in the next step.

  12. Next examine the Dockerfile in the build directory:

    cat build/Dockerfile
    Sample Output
    FROM quay.io/operator-framework/ansible-operator:v0.6.0

COPY roles/ ${HOME}/roles/
COPY watches.yaml ${HOME}/watches.yaml
COPY playbook.yml ${HOME}/playbook.yml

    Notice the source image for the container image that is being built. Also notice that all the Dockerfile does is to copy the roles, playbook and watches files into the container image.

    The ansible-operator container image already contains the logic that an Operator needs and just uses the playbook to set up the application.

    The playbook needs to be idempotent (meaning it must not fail when run a second time) because the operator keeps running the playbook periodically to ensure all necessary objects are in place.

    While the roles in this example support removing the applications (by setting state=absent) this is not necessary for roles in Ansible Operators. When the Ansible Operator creates Kubernetes Objects it injects ownerReference fields into all created objects. Therefore deleting the CR (Gogs in our example) will delete all associated objects.

  13. Before you can build with Docker you need to enable and start the Docker daemon on your VM (and add your user to the group that’s allowed to manipulate Docker).

    sudo systemctl enable docker
sudo systemctl start docker

# Replace the user below with your user id
sudo usermod -aG docker wkulhane-redhat.com

  14. After this you will need to log out of your VM and back in the new permissions to be in effect.

  15. Now build your operator container image. In order to build and push the container image to a registry you will need a user id at that registry. You could use either Docker Hub or Quay.

  16. For the purposes of this lab it is recommended that you create an account at https://quay.io if you do not already have one.

    The create account link is a bit hidden. Click on Sign In in the top right corner to see the login screen - which also has the link to Create Account.

    Once you have created your Quay account make sure to also create a public repository in Quay called gogs-operator. This will be where you will push your container image. Ask the instructor if you need any help with this.

    Make sure to select the public radio button on the Create New Repository page.

  17. Once you have your userid and repository in Quay you will need to log into Quay for the subsequent docker push command to succeed.

    docker login -u <your user> quay.io

  18. Now use the Operator SDK to build the container image and tag it as quay.io/<your quay id>/gogs-operator:v0.0.1. Don’t forget to change back into the operator directory if you needed to log back into your VM.

  19. Now push the container image to Quay:

    docker push quay.io/${QUAY_ID}/gogs-operator:v0.0.1

  20. The final piece is to modify the operator deployment definition (deploy/operator.yaml) to use the image that you just pushed and set the imagePullPolicy to IfNotPresent.

3.2. Deploy the Operator

Operators can be deployed on a cluster-wide basis or on a namespace/project basis. In this lab you are deploying your operator in your own project.

  1. Create the operator project (make sure to replace xyz with your initials):

    oc new-project xyz-gogs-operator --display-name="Gogs"
    You are using xyz-gogs-operator because you already have another project with name xyz-gogs on this cluster. Ordinarily if you used the one operator per namespace approach you would just name the project after the managed resource, not after the Operator.

  2. Now deploy one by one the artifacts in the deploy subdirectory.

  3. The first one to deploy is the service_account that our Operator will use.

    oc create -f ./deploy/service_account.yaml

  4. Next are the Role and RoleBinding that grant the correct permissions to the operator.

    The Operator SDK will create a skeleton Role file for you. However depending on what the operator needs to create or manipulate it will most likely be necessary to customize the Role.

    The Gogs Operator deploys some OpenShift-specific objects like a Route. Routes are defined in the route.openshift.io API group. So routes needs to be added to the list of resources that the Operator needs permissions for.

    Edit the Role Definition (vim ./deploy/role.yaml) to include routes as an object that the operator needs access to. Add another apiGroup for route.openshift.io. The only object in that apiGroup is routes.

    You will need to remove events because you don’t have permission to manipulate events.

    The Gogs operator does not use any of the built-in controllers. Therefore you may delete the entire apiGroup apps from the role (leave the second occurrance that grants permissions to deployments/finalizers).

    Replace the "*" resource in the gpte.opentlc.com API Group with gogs and gogs/status

    Unlike basic Kubernetes the OpenShift Security settings require the verbs to be listed rather than a '*' placeholder. So you will need to add the verbs create, update, delete, get, list, watch, patch to the "", route.openshift.io and gpte.opentlc.com API Groups.

    And finally Operator Metering is not installed on this cluster. Which means that you need to remove the monitoring.coreos.com API Group.

  5. Now create the Role and RoleBinding (granting the permissions in the role to the service account that the Operator uses):

    oc create -f ./deploy/role.yaml
oc create -f ./deploy/role_binding.yaml

    oc create -f ./deploy/operator.yaml

4. Using the Operator to create a Gogs Server

  1. To use an Operator to deploy an application it is necessary to create a Custom Resource - which is a Kubernetes Object. When the custom resource gets created in a project that the Operator is watching the operator then springs into action and creates the objects that make up that application based on the configuration options in the custom resource.

  2. The Gogs Operator understands the following Configuration Options:

    Name

    Default

    Required

    postgresqlVolumeSize

    1Gi

    No

    gogsVolumeSize

    1Gi

    No

    gogsSsl

    False

    No

    In Kubernetes the convention for variable names is to use CamelCase while in Ansible it is snake_case. Therefore the Custom Resource needs the variable nambes to be in CamelCase - while the Ansible Playbooks and Roles expect them in Snake Case. The Ansible Operator base image handles this translation automatically.

  3. The generated file deploy/crds/gpte_v1alpha1_gogs_cr.yaml includes a skeleton custom resource - but of course the operator SDK generator can not know which variables the playbooks in the Ansible Operator understand. So you will need to customize the CR.

    When writing an Operator you will need to consider that there may be more than one instance of Operator created objects in a project. In the Gogs example that means there may be more than one PostgreSQL Database and more than one Gogs Server.

    Any objects that get created need to have a unique name - usually this is achieved by using the name of the custom resource as a prefix to any created object. The roles you are using are built to satisfy these requirements.

  4. Create a Custom Resource to deploy a Gogs Server in your project.

    echo "apiVersion: gpte.opentlc.com/v1alpha1
kind: Gogs
metadata:
  name: gogs-server
spec:
  postgresqlVolumeSize: 4Gi
  gogsVolumeSize: 4Gi
  gogsSsl: True" > $HOME/gogs-operator/gogs-server.yaml

  5. Now create the Gogs Server:

    oc create -f $HOME/gogs-operator/gogs-server.yaml

  6. In one Shell Window watch as the pods come up:

    watch oc get pod

5. Deleting the Gogs Servers

Deleting an object created by an operator is as easy as deleting the custom resource. Operators who follow best practices set an ownerReference field in all created objects to enable cascading deletions. The Ansible Operator watches for Kubernetes API calls to create resources and adds the ownerReference field to every created resource after it’s been successfully created.

  1. Examine the Owner Reference of the created pods. First retrieve a list of pods.

6. Deleting the Gogs Operator

  1. To delete the Gogs operator simply delete the deployment that hosts the operator, the Role and RoleBinding as well as the Service Account.

    oc delete deployment gogs-operator
oc delete rolebinding gogs-operator
oc delete role gogs-operator
oc delete sa gogs-operator

  2. Finally you can delete the project (which would have also deleted all of the above…)

    oc delete project xyz-gogs-operator

  3. An Administrator will need to delete the ClusterRole and Custom Resource Definition.
